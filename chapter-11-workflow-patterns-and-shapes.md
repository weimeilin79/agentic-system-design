## The wall of red

Put your agent's architecture on a whiteboard, every box, every arrow. Take a red marker and shade in every box whose output you cannot predict, every place a model decides something. Step back and look.

On most first drafts, nearly the whole board is red. Somewhere along the way "build an agent" quietly became "let the model do every step," and now every box is a place the system can fail differently tomorrow than it did today. But look closely at what is inside them. Here is a box that looks up an order status. Here is one that converts a date, one that adds three numbers, one that picks a handler from a table of five. None of those needs a reasoning model. Each has exactly one right answer, and a line of ordinary code produces it perfectly, instantly, for a fraction of a cent, the same way every time. You painted them red out of habit.

That is the most useful realization in this part of the book, and it runs against every instinct the hype trains into you: the most agentic thing you can do is use less agent. This chapter is a single act performed on that whiteboard, turning back to black every box that does not need to be red, and then wiring the boxes that stay red into shapes that keep them from hurting you. Where you draw that line, and how you build around the red that remains, is the most consequential decision in the system.

## Why you want black

Every box you turn from red to black buys four things a model call cannot give you. A black box is **testable**: same input, same output, so an assertion you write today still holds next month. It is **auditable**: when it misbehaves you point at the line that did it, instead of squinting at a reasoning trace. It is **cheap and fast**: a lookup answers in a millisecond for almost nothing, where a model call to do the same work costs a thousand times more and can still be wrong. And it is **reliable** in a precise sense, its failures are the ones forty years of software engineering already knows how to catch and contain. Black is not the boring part you tolerate. Black is the part you can stand on.

So the default inverts from where the hype puts it. You do not start with a model and carve out what you can automate. You start with everything deterministic and let red back in only where black genuinely cannot do the job: interpreting a vague request, judging tone, choosing which of forty runbooks fits an incident nobody has seen. The test is one question asked of every box. Does this have a knowable right answer? Then it is black. Does it need judgment over something you cannot enumerate? Then it is red, and it needs an architecture around it.

## The one rule every shape obeys

Before the shapes, the rule they all serve, because it is about consequences, not cleverness. A model may **interpret, classify, summarize, extract, draft, rank, propose, and route**. Each of those produces a *suggestion* something else can check. A model may **not approve, pay, provision, submit, delete, or write to a system of record**, not without a deterministic check between its suggestion and the act.

The model produces proposals; code produces consequences. A proposal is allowed to be wrong, because something catches it. A consequence is not, so a probabilistic box must never be the last one before a consequence. Every architecture below is a different way of arranging red and black boxes so this stays true, and each one puts a deterministic **seam-check** at the point where a red box hands to a black one. The seam-check is always drawn from the same short menu:

- **Constrain the shape.** Force the red box's output into a schema at the decoder, so malformed output cannot be emitted at all. Grammar-constrained decoding makes this native. It guarantees well-formed, not correct.
- **Validate.** Check the output's meaning, not just its shape: does this ID resolve, is this value in range, is this action permitted. On failure, hand the red box back its own output and the specific error and let it retry, which it is good at.
- **Verify.** For high stakes, add an independent check: a second model that critiques, a rule that must also pass, or agreement across several samples.
- **Abstain.** Let the red box decline. Below a confidence threshold or outside a known-good domain, it routes to a human instead of guessing (Chapter 8).
- **Authorize.** Before any consequential action commits, a deterministic policy layer checks it against the rules and can refuse (Chapter 19).

Which checks you place, and how many, is set by the stakes of that particular seam.

One question sorts the shapes before you meet them: where does the model sit? In some shapes the model is at the **edge**, a bounded box that code calls into for a judgment and then takes back control. Code drives; the model advises. The Chain, the Router, and Parallel and Vote all work this way, and they are the safer, more predictable half, because determinism owns the control flow. In others the model is at the **core**: it decides what happens next, and code executes what it chose. The model drives; code carries out and checks. The Orchestrator-Worker, the Evaluator-Optimizer, and the Autonomous Loop live here, and they are more capable and less predictable, because you have handed the model the steering. That axis is the same as "how much red control you are willing to hold," and the shapes below run along it, from fully edge to fully core. One shape at the end sits outside the axis entirely, because it has no model in it at all.

## The Chain

*A fixed pipeline of steps, some black, some red, output flowing straight through, with a seam-check between each step.*

The simplest architecture, and the one to default to. You know the steps in advance and their order never changes, so you write them as a straight line: parse the document (black), extract the key facts (red), validate them against the record (black), draft the summary (red), check it against a schema (black), write it out (black). Control flow is entirely deterministic. The model is used at two specific steps and nowhere else, and a validate seam-check sits after each red step so a bad extraction never reaches the draft.

**Tradeoffs.** Maximally predictable, testable, and debuggable, because the path is fixed and you can assert on every intermediate result. The cost is rigidity: a Chain only handles the cases its fixed sequence anticipates, so it is wrong for a task whose steps depend on what the input turns out to be. **Where the seam-check goes:** after every red step, a constrain-plus-validate, since the next step depends on this one being well-formed and true. **In the wild:** this is Anthropic's prompt chaining, and it is the backbone of most document-processing and report-generation pipelines, the workhorse that quietly does the majority of the work in systems everyone calls "agents."

## The Router

*A red box reads the input and names its category; a deterministic table dispatches to the matching specialist.*

When a system handles several distinct kinds of request that each want different handling, put classification where models shine and dispatch where code shines. The red box reads the messy incoming request and emits one label from a fixed, known set. It does not perform the routing itself. A deterministic table maps the label to a handler, and each handler is its own downstream shape, often a Chain of its own. The intelligence chooses the door; the code opens it.

**Tradeoffs.** Clean separation of concerns, and each specialist path can be simpler and better-tested than one monster prompt trying to cover everything. It adds a hop of latency and one more red box to evaluate. **Where the seam-check goes:** right on the label, a constrain-plus-validate that forces the output to be exactly one member of the allowed set, so the model cannot invent a destination that does not exist, and an abstain when it is unsure so an ambiguous request goes to a human instead of a wrong handler. **In the wild:** customer-service front doors (classify the ticket, route to the right flow), and Google's ADK builds this in directly as an LLM-routing agent that hands off to sub-agents.

## Parallel and Vote

*Run several red boxes at once, then let a black box aggregate their outputs.*

Two different needs share this shape. **Sectioning** splits a task into independent pieces that run in parallel (summarize each of ten documents, then combine), which buys speed. **Voting** runs the *same* question through the model several times and takes the answer they agree on, which buys confidence. Either way, the red boxes run concurrently and a deterministic aggregator, a majority vote, a sum, a merge, produces the single result. The aggregation is always black.

**Tradeoffs.** Sectioning cuts latency; voting turns a single unreliable call into a more reliable consensus, at a cost in tokens that multiplies with the number of samples. Reserve voting for the decisions that genuinely warrant paying three-to-five times for. **Where the seam-check goes:** in the aggregator itself, which is where verify lives, disagreement across the votes is the signal to lower confidence or escalate rather than pick one arbitrarily. **In the wild:** self-consistency is the standard trick for squeezing reliability out of a model on hard single-answer questions, and sectioned parallelism is how research and multi-document agents stay fast.

## The Orchestrator-Worker

*A red box plans: it decomposes a task into subtasks it decides at runtime, and a deterministic harness dispatches workers and synthesizes their results.*

This is the shape for a task whose steps cannot be known ahead of time, where the number and nature of the pieces depend on the specific input. A coding change is the canonical case: which files to touch, and how, is not something you can hardcode, so a planner red box works it out, emits a structured plan, and a black harness runs each step, feeding results back. It is your "reasoning core with deterministic execution": the model owns the *what*, code owns the *how*. Crucially, the plan is validated as a whole before anything runs, and the workers execute on rails.

**Tradeoffs.** Enormous flexibility, the reason serious coding and research agents use it, at the price of the most non-determinism of any shape so far, since the plan itself is generated. It is also where cost and latency climb. **Where the seam-check goes:** on the plan, before execution, validate that every step is legal and within policy and show it to a human if the stakes call for it (the Deterministic Executor move, plan then commit); and authorize each worker's consequential actions individually, since a validated plan can still contain a step that should not run unattended. **In the wild:** this is Anthropic's orchestrator-workers pattern and Andrew Ng's planning pattern under another name, and it is the backbone of the coding agents. Google's ADK expresses it as a deterministic workflow agent wrapping model-driven sub-agents.

## The Evaluator-Optimizer

*A generator red box produces a draft; an independent critic red box scores it against criteria; the generator revises, looping until the critic passes or a deterministic limit trips.*

When first-pass quality is not good enough and there is a clear standard to judge against, put a critic in the loop. One box generates, a second, deliberately separate box evaluates against explicit criteria and hands back specific feedback, and the generator revises. The two roles are kept apart on purpose, because a box grading its own work shares its own blind spots. The loop is closed by a black box: a passing score, or a hard cap on iterations so it cannot spin forever (Chapter 10's termination rules apply directly).

**Tradeoffs.** Real gains in output quality for tasks with a checkable standard (a translation that must preserve meaning, code that must pass tests), at the cost of extra passes and the risk of a loop that never converges. **Where the seam-check goes:** the critic *is* a verify seam, and the deterministic termination check is the second, non-negotiable one, since two red boxes talking to each other with no black exit is exactly the runaway loop from the last chapter. **In the wild:** this is Anthropic's evaluator-optimizer, and the pattern behind agents that write, test, read the failures, and fix, where the test suite is the critic and the compiler is an honest judge.

## The Autonomous Loop

*The model owns the control flow. It decides each next action at runtime, and the architecture's whole job is to fence what it is allowed to do.*

The maximally-red shape. There is no predetermined path at all; the model observes, decides, acts, and repeats, choosing its own next move each turn. This is a real ReAct-style agent (Chapter 10), and it is the right shape only when the task is genuinely open and no amount of upfront structure fits. Because control itself is red, the fencing cannot live in the flow; it has to wrap every action. Bounded Authority does the heavy lifting here: the model can propose anything, and a deterministic authority layer decides what actually commits.

**Tradeoffs.** The most capable and the least predictable shape, the one that earns the warnings. You reach for it last, after a Chain, a Router, or an Orchestrator-Worker has genuinely failed to fit, because you are trading away most of what determinism bought you. **Where the seam-check goes:** everywhere, and hard. Authorize every consequential action, cap the loop and detect no-progress (Chapter 10), and keep the model's authority fully mediated so it can ask for anything and cause nothing directly. **In the wild:** the open-ended coding and computer-use agents run here, wrapped in exactly this fencing, and when the red boxes in this shape grow big enough to become their own agents, you have crossed into multi-agent orchestration, which is the next chapter.

## The Channel Adapter

*A deterministic adapter per input channel normalizes every arrival into one canonical request, and a renderer per channel formats the reply on the way out.*

This one wraps the others rather than competing with them. Real agents have many front doors: a webhook, a Slack message, an email, a cron sweep, each with its own protocol and its own idea of who is asking. Let that reach the red boxes and the model ends up parsing protocols, badly and by the token. Give each channel a small black adapter instead: authenticate the caller, parse the payload, emit one canonical request (sender, intent, attachments, origin), and turn anything that is not text into text at the door. Every shape above then sees one format and never learns that Slack exists.

**Tradeoffs.** One agent to build, evaluate, and fence, and a new channel costs one small adapter rather than a change to the agent, at the price of a canonical schema every channel must fit, which pulls toward a lowest common denominator. **Where the seam-check goes:** at the door, and its job is provenance, because an authenticated message from a colleague and an email from a stranger are not the same input. The adapter verifies signatures, resolves the sender, and stamps each request with origin and trust for the authorize check downstream. Then the rule people skip: channel content is data, never instructions, so reading a message can never by itself authorize a tool call. **In the wild:** OpenClaw collapses a dozen-plus platforms into one message object before its loop runs, voice transcribed at the door, and a security review of that layer found dozens of advisories concentrated in it, each adapter written independently with no shared identity check.

## Choosing a shape

The shapes are ordered by how much control you hand the model, from fully at the edge to fully at the core, and that order is the decision procedure. Start at the top and stop at the first shape that fits. A fixed sequence is a Chain. Distinct categories is a Router. Independent pieces or a need for consensus is Parallel and Vote. Unpredictable decomposition is an Orchestrator-Worker. A quality bar to iterate against is an Evaluator-Optimizer. Genuinely open control is an Autonomous Loop. Most real systems nest several: a Router at the front dispatching to Chains, one of which uses an Evaluator-Optimizer on its output. Reach down the list only when the shape above genuinely does not fit, because every step down trades predictability for flexibility, and you pay for that trade in every property from the "why you want black" list.

## In the wild

The vocabulary of these shapes is now shared across the industry, and drawing the deterministic/non-deterministic line inside them is the central architectural claim every platform makes, in its own words. Google's framing is the plainest, deterministic business rules paired with probabilistic reasoning, and ADK ships the shapes as primitives: its workflow agents (sequential, parallel, loop) are deterministic by construction, and you drop model-driven sub-agents into the boxes that need judgment. The Chain, the Router, and the Orchestrator-Worker are all expressible directly, with the graph black and the boxes inside it red or black as you choose. That is this chapter drawn in an API.

Everyone else names the same shapes. Anthropic's *Building Effective Agents* is the reference vocabulary, prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer, with the standing advice to reach for the simplest shape that works and treat multi-agent as a last resort. Microsoft's Agent Framework hard-separates Workflows (deterministic graphs) from Agents (probabilistic boxes), which is the black-skeleton-with-red-slots picture as a product. AWS makes deterministic policy its own AgentCore component, the authorize seam-check as infrastructure. The convergence is the tell: independent teams keep arriving at the same handful of shapes and the same rule about where the model may sit inside them. Different nouns, one line drawn in the same place.

## When not to over-shape

Two edges. On the high-assurance side, some paths tolerate no model variance at all, a regulated calculation, a safety interlock, a legally mandated step, and the move is to keep them fully black, with no red box in the path even for interpretation. On the other side, do not impose a heavy shape on a task that does not need one: if a single well-prompted call or a three-step Chain does the job, a Router-of-Orchestrators is complexity you will pay for in latency, cost, and debugging with nothing to show. The shapes are tools for controlling non-determinism you actually have, not a ladder to climb for its own sake.

## Which phase

This is the **loop** examined at its seam, the exact boundary where the deterministic harness meets the probabilistic model. Every shape here is a way of arranging that boundary. Everything after assumes the line has been drawn: multi-agent orchestration is what happens when the red boxes in an Orchestrator-Worker or an Autonomous Loop grow big enough to become agents in their own right, and validation and self-improvement are about watching and tightening the red you chose to keep. The whole rest of the book lives on one side of this line or the other, which is why where you draw it is the most consequential thing in the part.
