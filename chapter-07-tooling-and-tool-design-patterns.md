## You already know how to build an API. That's the problem.

In Chapter 2, tools were the hands and feet: the part of the agent that reaches out of the chat window and does something in the world. Query a database, file a ticket, restart a service. Without them an agent is a very articulate observer. With them it can act, and everything that makes agents useful and everything that makes them dangerous starts here.

So you sit down to build one, and your instincts kick in, and your instincts are mostly wrong. Because you have built APIs before, and a tool looks exactly like an API, and it is not one.

Here is the distinction the whole chapter turns on. An API is a contract between two deterministic systems. You call `getUser(123)`, you get user 123, formatted the same way every time, and the program on the other end handles it the same way every time. A tool is a contract between a deterministic system and a *reasoner that might do something different every run*. The caller reads your parameter names and guesses what they mean. It reads your response token by token and spends its limited attention doing so. It reads your error message and tries to recover, or doesn't. Design accordingly, and the counterintuitive rules below stop being counterintuitive.

## How tools break

Tools fail in recognizable ways, and naming them tells you which pattern to reach for.

- **Wrong granularity:** you wrapped your REST API one endpoint to one tool, so the agent has to make six calls and do the joining itself.
- **Token flooding:** the tool returns everything, and the agent burns its context reading a thousand rows to find one.
- **Ambiguity:** a parameter called `user` or a date with no format, and the agent guesses wrong.
- **Silent failure:** the tool errors with a stack trace the agent can't act on, so it retries the same broken call forever.
- **Too many tools:** dozens of servers, hundreds of overlapping tools, and the agent can't tell which one to pick.

Every pattern here is an answer to one of those.

## Designing a tool

**Workflow Tools.** *Build a few high-signal tools around real tasks rather than a wrapper per API call.* Mirroring your API one-to-one is a reasonable way to start, and often the quickest path to something that runs, but treat it as a starting point to evolve from, not the destination. A model handed twenty low-level CRUD tools has to plan the orchestration itself, badly. Instead, build the handful of tools that match the workflows the agent actually performs, and let each one consolidate several underlying calls behind the scenes. Prefer `search_customers(query)` over `list_all_customers()` that returns everyone. One good tool that does the whole job beats six primitives the model has to assemble. Start with a few, measure, and add only when a real task needs one.

**High-Signal Returns.** *The response is context, so return what informs the next action and drop the rest.* Every token a tool returns is a token the agent can't spend on reasoning, so a tool that dumps raw records is actively hostile. Strip the low-level identifiers the model will never use (the `uuid`, the `mime_type`, the `256px_thumbnail_url`) and return the fields that actually drive a decision (the name, the status, the amount). Build in pagination, filtering, and truncation with sane defaults, and consider a `concise` versus `detailed` mode, teams have cut tool-response tokens by well over half just by letting the agent ask for less. The address-book rule says it best: don't hand the model the whole book to read page by page, hand it the one entry it asked for.

**The Description as Prompt.** *Write the tool's name, parameters, and docs as if onboarding a new hire, because you are.* The tool definition is loaded straight into the agent's context, so it is prompt engineering whether you treat it as such or not. Name parameters unambiguously (`user_id`, not `user`), state formats explicitly (a date is `yyyy-mm-dd`, say so), and enforce it all with a strict schema so a malformed call fails fast instead of silently. Make the implicit explicit: the query syntax you take for granted, the niche term, the relationship between two resources. This is not busywork. Studies of real MCP servers find that vague descriptions are one of the biggest drags on agent performance, and that fixing a single fuzzy parameter can change the agent's whole strategy for the better.

**Teaching Errors.** *An error message is another prompt: tell the agent how to fix the call, not just that it failed.* When a tool returns a raw traceback or an opaque code, the agent has nothing to act on and will often retry the identical broken call. Instead, write errors that steer: "start_date must be yyyy-mm-dd, you sent 03/14/2026," or "no results, try a broader query or add a region filter." A good error turns a dead end into a corrected next step. This is the same "keep the errors visible" idea from Chapter 5, now designed into the tool itself.

## Surviving many tools

One agent with five tools is a design problem. One agent with eight tool servers and two hundred tools is an infrastructure problem, and no amount of careful schema-writing will fix it. What matters now is how many tools the agent is shown at all, and who gets to decide.

**Namespacing.** *Group related tools under a common prefix so the agent can tell them apart.* Once an agent has access to dozens of servers, overlapping and vaguely named tools become a selection problem: which `search` did you mean? Namespacing by service and resource (`asana_search`, `asana_projects_search`, `jira_search`) draws clean boundaries the model can navigate. It sounds cosmetic and isn't, the prefixing scheme measurably changes tool-selection accuracy.

**Tool Placement.** *Standardize on MCP, then choose whether a tool lives local to one agent, in a shared library, or on its own server.* Before MCP, every tool integration was a one-off: new glue for every model, every framework, every team, forever. The Model Context Protocol standardized the contract, so a server you write once works across every major agent client. Released in late 2024, it is now the de facto standard, governed alongside A2A under a neutral foundation. Use it.

But settling the protocol only sharpens the more interesting question: *where does the tool actually live?* Three answers, and they are a real trade.

A tool can be a **utility function inside one agent**, if only that agent will ever call it. Fastest to write, no deployment, no coordination... and the moment a second agent needs it, you copy-paste, and now there are two of them drifting apart.

It can live in a **shared library**, a package several agents import. Good for logic that is genuinely common, and it makes reuse cheap, but every consumer now pins a version, and upgrading the library means a coordinated redeploy of everything that imports it. You have traded runtime coupling for release coupling.

Or it can be a **tool server**, deployed and versioned on its own, with agents connecting over MCP. This is the right default for anything shared across teams or agents. The tool ships on its own cadence, one team owns it, one place gets fixed when it breaks, and the agents don't redeploy when it changes. The cost is that you now run a service, with the network hop, the availability, and the operational weight that implies.

Versioning is where sharing gets real. A tool used by one agent can change freely, because there is exactly one caller to fix. A shared tool cannot: change a parameter's meaning and every agent depending on it silently starts doing something subtly different, which is a far worse failure than a compile error, because a reasoner will simply adapt and carry on producing plausible nonsense. Version the interface, additively where you can, and keep old versions alive while callers migrate. Agents are more forgiving than compilers about a fuzzy parameter, which is exactly why nothing will tell you when you break them.

**The Tool Gateway.** *Aggregate your tool servers behind one governed endpoint that decides which tools each agent can see and call.* Wire eight tool servers directly into an agent and you have signed up for eight auth flows, eight sets of logs, eight places a tool name can collide, and eight third parties whose tool descriptions now land in your model's context. A gateway is a single front door in front of all of them. The agent connects once, with one credential. The gateway authenticates the caller, decides which tools that identity may see, routes the call, and writes the log.

Deciding what an agent may *see* is the lever. A gateway issues each consumer its own scoped tool surface, an allow-list per agent, per team, per environment, so an incident agent is presented with the three remediation tools it needs rather than the two hundred the organization happens to run. Its context stays small because it was shown fewer tools, whatever the total number your organization runs. Meanwhile every call flows through one place, which makes the gateway the natural home for authentication, rate limits, audit records, and traces.

It's a better security implementation at the same time. A third-party server's tool descriptions load straight into the model's context, which means a malicious description is a prompt injection with a clean delivery path. The attack is called tool poisoning, and the gateway is the chokepoint where you can scan for it before anything reaches the model.

One distinction worth keeping straight: a gateway is runtime infrastructure, a registry is a discovery index (Chapter 17). They solve different problems, and most teams end up with both, discovering a server through a registry and then putting it behind a gateway.

**The Guardrailed Boundary.** *Put every tool call through a checkpoint that authorizes, filters, and audits before it executes.* A tool is where the agent touches the real world, so it is exactly where control belongs. Route calls through a boundary that checks permissions, masks sensitive fields, fetches credentials at call time rather than baking them in, and writes an audit record, all before the action runs. Never let the model be your policy engine. This is a preview of Chapter 20; the point here is that the tool layer is the natural home for it.

**Tool Evals.** *Measure the agent's ability to use the tool, because you can't eyeball a non-deterministic caller.* This is the one kind of software whose caller behaves differently every run, so "it works when I try it" proves nothing. Build a set of realistic tasks, run the agent against them, and watch not just success but how many calls it took and how many tokens it burned. Tools rewritten by an agent against an eval set have beaten the human-written originals on both task completion and token use. A tool without an evaluation is a guess.

**Code Execution.** *Let the agent write code that calls tools as functions instead of round-tripping every call through the model. Reach for it when the tools aren't yours to fix.* Bloated definitions and oversized returns are better solved by building fewer, sharper tools. What good design can't fix is a toolset you don't own: connect to a handful of third-party servers and you inherit hundreds of definitions and whatever payloads they return. Nor can you anticipate every composition, or stop data from transiting the window when tool A's output feeds tool B and the model retypes the payload into the next call.

Presenting tools as a code API answers all three. They become importable functions, so the agent writes a short script that imports only what it needs, definitions load on demand, intermediate results stay as variables rather than tokens, and only what the script prints returns to context. Filtering and looping happen in code rather than one model turn at a time.

This only works inside a sandbox with real controls (Chapter 18), since you are executing model-written code. That makes it a natural fit for local development, and something to introduce deliberately in production. Several gateways now run code execution behind their own boundary for exactly that reason.

## The tool-count ceiling

There is a soft ceiling, somewhere around twenty to fifty tools, past which selection accuracy falls off no matter how well you namespace. When you hit it, you have four honest moves, and adding a fifty-first tool is not one of them. Consolidate several tools into fewer, workflow-shaped ones, which is the first answer and usually the right one. Filter what the agent sees at a gateway, so the tools exist but this agent isn't shown them. Split the work across sub-agents (Chapter 13), each with its own focused, smaller toolset. Or, when the tools aren't yours to consolidate, switch to code execution inside a sandbox, where the agent imports tools on demand instead of holding all of them in context at once. Dynamic tool retrieval, letting the agent search for the tool it needs the way it searches for a document, is an active and not-yet-solved research area, so for now the reliable answer is fewer tools in front of any one agent, not more.

## In the wild

Google's MCP Toolbox for Databases is the tool-server argument made concrete. Rather than embedding connection handling, pooling, parameterized queries, and error handling inside each agent, you declare the tools in a YAML file and the Toolbox serves them as an MCP server, handling auth and observability. The decoupling is the point: you change a query by editing the config and restarting the Toolbox, without touching agent code, and the same tools work across ADK, LangGraph, LlamaIndex, or any MCP-compatible client. One tool definition, many agents, versioned independently. Google's agent platform pairs that with an Agent Gateway, Agent Registry, and Agent Identity, which is the gateway-and-registry split of this chapter shipped as products. Enterprises are building on exactly this seam: L'Oréal connects its agents through MCP to its internal data platform and operational systems rather than wiring each integration by hand.

The strongest evidence that tools are prompts, not APIs, comes from tool-optimization work at Anthropic: when Claude was allowed to rewrite the Slack and Asana MCP tools against an evaluation set, the agent-rewritten versions beat the human-written originals on task completion and token efficiency. And Spotify's Honk (Chapter 25) exposes its build-and-verification steps to a coding agent as abstract tools over MCP rather than raw shell access, which is what lets the agent act on thousands of repositories without bespoke wiring for each.

## When not to reach for these

If your agent calls one stable, internal service with a fixed contract, most of this is overkill. A single well-described function does not need namespacing, MCP, or a code-execution layer, and wrapping a deterministic, unchanging integration in the full apparatus buys you complexity for nothing. The patterns here earn their keep when the agent has real choices to make among real tools, when the toolset grows, spans teams, or changes underneath you. For a narrow, fixed capability, describe it well, guardrail it, and stop.

## Which phase

Tools are the **harness**: the runtime capabilities the agent acts through, the part of the system that survives a model swap. But every tool is also paid for in **context**, because its definition and its every response live in the window and compete for the same attention budget as everything in Chapter 5. That dual citizenship is the whole tension of tool design, a tool is only as good as the job it does *and* the context it costs, and the best ones are ruthless about both.
